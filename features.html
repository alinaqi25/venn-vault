What's inside

Every feature has a reason.

Not just what it does, but why it was built that way. Each card below covers both the implementation and the thinking behind it.

⚙

Zero-Framework HTTP Server

Backend Core

Built on Node's native http module with no framework. Routes are matched with if-blocks on method and URL. The body parser reads the raw request stream and parses it as JSON. Writing this by hand is what made Express's abstractions actually make sense to me, rather than just being things I import and trust.

🔑

bcrypt Password Hashing

Security

Passwords are hashed with bcrypt at cost factor 10 before anything touches the database. bcrypt is intentionally slow, which is the point: it makes brute-force attacks expensive. On login, bcrypt.compare() handles the check. The original password is never stored and never recoverable.

🪙

JWT-Based Authentication

Security

After login, the server signs a JWT and sets it as an HttpOnly, SameSite=Lax cookie expiring in 10 minutes. HttpOnly means the token is inaccessible to JavaScript on the page entirely, which removes the XSS token theft vector that comes with localStorage. No session store, no server-side state.

🔮

Database-Generated Identifiers

Database

Account numbers are a GENERATED ALWAYS AS computed column in PostgreSQL, derived from a sequence, so they follow a VV-XXXXXXXXXX format with zero application-side logic. Wallet IDs use GENERATED BY DEFAULT AS IDENTITY starting at a large offset so IDs are never guessably low.

🗄

PostgreSQL Relational Ledger

Database

Three tables: users, wallets, transactions. Balances live on wallets, not on users, so a user can eventually have multiple wallets without a schema change. Every transaction records sender and receiver wallet references, giving a complete, queryable audit trail of every PKR movement.

💸

Atomic Fund Transfers

Banking

Transfers run inside a PostgreSQL transaction with FOR UPDATE row locks on both wallets before touching either balance. Either the debit, credit, and transaction record all commit together, or none of them do. A transfer that partially succeeds would be a bug. This one can't.

📊

Transaction History

Banking

A single query joins transactions to wallets to users on both the sender and receiver side, returning names and account numbers alongside each transaction row. No N+1 lookups. The admin panel runs the same query for any user in the system.

🛡

Admin Control Panel

Admin

Admin is a seeded database user with account_type = 'admin', not a hardcoded password check or env flag. Login issues a separate adminToken JWT cookie. Every admin route verifies the token and checks the role claim. From the panel: view all accounts, freeze or unfreeze users, soft-delete accounts, and inspect any transaction history.

🧪

Isolated Transaction Layer

Architecture

Payment processing is deliberately out of scope. The goal was to focus on routing, database concurrency, and auth correctness, not third-party API integration. Separating those concerns means the transaction layer can be tested thoroughly on its own, which is a real architectural pattern.

🍪

Dual-Cookie Session Design

Security

User and admin sessions are completely separate cookies: token for users, adminToken for admin. Both are HttpOnly and SameSite=Lax, set and cleared manually in the server response headers. No cookie library, no framework middleware handling this quietly.
