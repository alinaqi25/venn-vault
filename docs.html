Introduction

Venn-Vault is a full-stack banking API built in pure Node.js with no framework. The goal was to understand what actually happens inside a web server before reaching for abstractions like Express.

It covers user registration and login, deposits, withdrawals, fund transfers, transaction history, and an admin control panel. The backend is a single Node.js process. The database is PostgreSQL. There are three npm packages: pg, bcrypt, and jsonwebtoken.

Note: Payment processing is intentionally out of scope. The transaction layer accepts arbitrary amounts without real payment validation so the focus stays on routing, database writes, and auth correctness.

Why Vanilla Node?

I wanted to know what Express is actually doing. The answer is quite a lot: body parsing, content negotiation, routing, error handling, response helpers. By doing all of that manually, you stop treating those things as magic and start treating them as solved problems you understand.

This isn't an argument against frameworks. Express is great and I'll use it in the next project. But I'd rather know what I'm abstracting over before I abstract over it.

Tech Stack

  • Runtime: Node.js, native http, fs, path, url modules
  • Database: PostgreSQL via the pg client
  • Password Hashing: bcrypt
  • Authentication: jsonwebtoken, stateless JWT sessions via HttpOnly cookies
  • Config: dotenv
  • Frontend: HTML, CSS, Vanilla JavaScript (no build step)

Prerequisites

Make sure the following are installed on your machine before continuing:

  • Node.js (latest LTS recommended)
  • PostgreSQL (version 14 or higher)
  • A terminal with git available

Installation

1. Clone the repository

git clone https://github.com/alinaqi25/venn-vault.git
cd venn-vault

2. Install dependencies

The project is deliberately minimal. The only npm packages are pg, bcrypt, and jsonwebtoken. 

npm install

3. Set up the PostgreSQL database

Create a new database and run the schema file. This creates the users, wallets, and transactions tables along with indexes. 

psql -U your_postgres_user -c "CREATE DATABASE vennvault;"
psql -U your_postgres_user -d vennvault -f schema.sql

Environment Variables

Create a .env file in the project root. The server reads these at startup. It will not start without them. 

DB_USER=your_postgres_user
DB_PASSWORD=your_postgres_password
DB_HOST=localhost
DB_PORT=5432
DB_NAME=vennvault

JWT_SECRET=your_super_secret_jwt_key
ALLOWED_ORIGIN=http://localhost:8080

Warning: Never commit your .env file. The .gitignore already excludes it, but double-check before pushing to a public repository.

Running the Server

npm start

The server starts on http://localhost:8080. The port is hardcoded to 8080 in the server. Static files are served from the same process, so navigating to the root URL opens the app directly.

Custom Router

Routing is handled with a chain of if-blocks inside the http.createServer callback. On each incoming request, the server checks request.method and request.url against each condition in order. If nothing matches, a 404 is returned. Static files are served first for any non-/api/ GET request, falling back to appending .html for extension-less paths.

Dynamic path segments (e.g. /api/admin/users/:id/freeze) are matched by checking url.startsWith() and url.endsWith(), then the ID is extracted by splitting the URL on /. 

// Simplified view of the routing approach
if (url === '/api/auth/login' && method === 'POST') { ... }
if (url === '/api/wallet/deposit' && method === 'POST') { ... }
if (url.startsWith('/api/admin/users/') && url.endsWith('/freeze') && method === 'POST') { ... }

Auth Flow

Registration

The user provides a name, email, and password. The server hashes the password with bcrypt at cost 10, inserts the user record into PostgreSQL, then creates a linked wallet row with a starting balance of 0 PKR. Account numbers are generated by the database as a computed column. On success, the user is immediately signed in and a JWT cookie is set.

Login

The user provides their account number and password. Soft-deleted accounts are rejected first. Then the server fetches the hash and runs bcrypt.compare(). On success, a JWT containing the account number is signed and set as an HttpOnly, SameSite=Lax cookie expiring in 10 minutes.

Protected Routes

Each protected route manually reads the token cookie from the request headers, verifies the JWT signature against JWT_SECRET, and looks up the user by account number. Requests with a missing, expired, or invalid token get a 401.

Database Schema

users

CREATE TABLE users (
  id             BIGSERIAL PRIMARY KEY,
  account_seq_id SERIAL UNIQUE,
  account_number VARCHAR(30) GENERATED ALWAYS AS
                   ('VV-' || (1075102638 + account_seq_id)::TEXT) STORED UNIQUE,
  name           VARCHAR(100) NOT NULL,
  email          VARCHAR(255) UNIQUE NOT NULL,
  password_hash  VARCHAR(255) NOT NULL,
  account_type   VARCHAR(10) NOT NULL DEFAULT 'user', -- 'user' | 'admin'
  is_frozen      BOOLEAN NOT NULL DEFAULT FALSE,
  create_time    TIMESTAMPTZ NOT NULL DEFAULT NOW(),
  is_deleted     BOOLEAN NOT NULL DEFAULT FALSE -- soft delete
);

wallets

CREATE TABLE wallets (
  id       BIGINT PRIMARY KEY GENERATED BY DEFAULT AS IDENTITY
             (START WITH 749201836),
  user_id  BIGINT NOT NULL REFERENCES users(id) ON DELETE RESTRICT,
  balance  NUMERIC(15,2) NOT NULL DEFAULT 0.00,
  currency VARCHAR(10) NOT NULL DEFAULT 'PKR'
);

transactions

CREATE TABLE transactions (
  id                 BIGSERIAL PRIMARY KEY,
  sender_wallet_id   BIGINT REFERENCES wallets(id) ON DELETE SET NULL,
  receiver_wallet_id BIGINT REFERENCES wallets(id) ON DELETE SET NULL,
  amount             NUMERIC(15,2) NOT NULL,
  transaction_type   VARCHAR(10) NOT NULL, -- 'DEPOSIT' | 'WITHDRAW' | 'TRANSFER'
  create_time        TIMESTAMPTZ NOT NULL DEFAULT NOW()
);

Auth Endpoints

POST /api/auth/register

Register a new user account. Also creates a linked PKR wallet and logs the user in immediately.

// Request body
{
  "name":     "Jane Doe",
  "email":    "jane@example.com",
  "password": "supersecret"
}

// Response 201  sets HttpOnly cookie: token=<jwt>
{
  "success": true,
  "message": "User registered and logged in successfully!"
}

POST /api/auth/login

Authenticate with an account number and password.

// Request body
{
  "accountNumber": "VV-1075102639",
  "password":      "supersecret"
}

// Response 200  sets HttpOnly cookie: token=<jwt>
{ "success": true, "message": "Login successful!" }

GET /api/auth/profile

Returns the authenticated user's account details and linked wallet. Requires a valid session cookie.

// Response 200
{
  "success": true,
  "user": {
    "id": 1,
    "accountNumber": "VV-1075102639",
    "name": "Jane Doe",
    "email": "jane@example.com",
    "role": "user",
    "dateCreated": "2026-05-01T10:00:00Z",
    "isFrozen": false
  },
  "wallet": {
    "id": 749201836,
    "userId": 1,
    "balance": "2500.00",
    "currency": "PKR"
  }
}

Account Endpoints

Account details are returned by GET /api/auth/profile, documented above in the Auth Endpoints section.

Transaction Endpoints

POST /api/wallet/deposit

// Request body
{ "amount": 500 }

// Response 200
{ "success": true, "message": "Deposit processed successfully.", "newBalance": 3000 }

POST /api/wallet/withdraw

// Request body
{ "amount": 200 }

// Response 200
{ "success": true, "message": "Withdrawal processed successfully.", "newBalance": 2800 }

POST /api/wallet/transfer

Recipient can be an email address or a numeric wallet ID. Runs inside a PostgreSQL transaction with row-level locks.

// Request body
{
  "recipient": "recipient@example.com", // email OR wallet ID
  "amount": 150
}

// Response 200
{ "success": true, "message": "Transfer processed successfully.", "newBalance": 2650 }

GET /api/wallet/transactions

Returns all transactions for the authenticated user, ordered by most recent. Each row includes sender and receiver names and account numbers.

// Response 200
{
  "success": true,
  "transactions": [
    {
      "transactionId": 1,
      "senderWalletId": null,
      "receiverWalletId": 749201836,
      "amount": "500.00",
      "transactionType": "DEPOSIT",
      "createTime": "2026-05-15T14:22:00Z",
      "senderName": null,
      "recvName": "Jane Doe",
      "recvAccNumber": "VV-1075102639"
    }
  ]
}

Admin Endpoints

Admin login issues a separate adminToken JWT cookie (30 minute expiry). All other admin routes verify this cookie and check that the token carries role: "admin" before proceeding.

POST /api/admin/login

Authenticate as admin using the seeded admin account password.

// Request body
{ "password": "your_admin_password" }

// Response 200  sets HttpOnly cookie: adminToken=<jwt>
{ "success": true, "message": "Admin login successful." }

GET /api/admin/users

Returns all non-admin users with their wallet balances and statuses.

GET /api/admin/users/:id/transactions

Returns the full transaction history for a specific user by their database ID.

POST /api/admin/users/:id/freeze

Freeze or unfreeze a user account. Pass { "frozen": true } to freeze, { "frozen": false } to unfreeze. Frozen users cannot log in or perform transactions.

POST /api/admin/users/:id/delete

Soft-deletes the user by setting is_deleted = TRUE. Transaction history is preserved. Deleted accounts cannot log in.
