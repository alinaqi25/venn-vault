Introduction

Venn-Vault is a full-stack online banking application with a deliberate emphasis on backend mechanics. It was built to explore what actually happens inside a web server — before Express abstracts it all away.

The result is a working banking API: user registration, login, deposits, withdrawals, fund transfers, and an admin control panel — all implemented from scratch in pure Node.js, backed by a PostgreSQL database.

Note: Venn-Vault uses a "sandbox economy" for transactions. Deposits and withdrawals accept arbitrary values without real payment validation — this is intentional, to isolate and test the routing and database layers.

Why Vanilla Node?

Most production Node.js apps reach for Express on day one. This project was built to ask what Express is actually doing. The answer turns out to be: quite a lot. By handling raw TCP streams, writing a custom body parser, matching routes with a hand-rolled pattern matcher, and manually setting response headers, you develop an understanding of the HTTP protocol that framework usage tends to obscure.

This is not an argument against frameworks — Express is excellent. It's an argument for understanding what you're abstracting before you abstract it.

Tech Stack

  • Runtime: Node.js — native http, crypto, fs modules
  • Database: PostgreSQL via the pg client
  • Password Hashing: bcrypt
  • Authentication: jsonwebtoken — stateless JWT sessions
  • Frontend: HTML, CSS, Vanilla JavaScript (no build step)

Prerequisites

Make sure the following are installed on your machine before continuing:

  • Node.js (latest LTS recommended)
  • PostgreSQL (version 14 or higher)
  • A terminal with git available

Installation

1. Clone the repository

git clone https://github.com/alinaqi25/venn-vault.git
cd venn-vault

2. Install dependencies

The project is deliberately minimal. The only npm packages are pg, bcrypt, and jsonwebtoken. 

npm install

3. Set up the PostgreSQL database

Create a new database in PostgreSQL and run the included SQL schema file to create the users and transactions tables. 

psql -U your_postgres_user -c "CREATE DATABASE venn_vault_db;"
psql -U your_postgres_user -d venn_vault_db -f schema.sql

Environment Variables

Create a .env file in the project root. The server reads these at startup — it will not start without them. 

PORT=3000

DB_USER=your_postgres_user
DB_PASSWORD=your_postgres_password
DB_HOST=localhost
DB_PORT=5432
DB_NAME=venn_vault_db

JWT_SECRET=your_super_secret_jwt_key
ADMIN_PASSWORD=your_admin_panel_password

Warning: Never commit your .env file. The .gitignore already excludes it, but double-check before pushing to a public repository.

Running the Server

npm start

The server starts on http://localhost:3000 by default, or whatever PORT you defined in .env. The frontend is served statically from the same process — navigate to the root URL to open the app.

Custom Router

The router is a plain JavaScript object that maps METHOD /path strings to handler functions. On each incoming request, the server reads req.method and req.url, strips query strings, and looks up the matching handler. If none is found, a 404 is returned.

Dynamic segments (e.g. /api/user/:id) are matched with a simple regex loop, and the extracted params are attached to the request object before the handler is called. 

// Simplified view of the routing logic
const routes = {
  'GET /api/account':        accountHandler.getAccount,
  'POST /api/deposit':       transactionHandler.deposit,
  'POST /api/transfer':      transactionHandler.transfer,
  'GET /api/transactions':   transactionHandler.history,
};

Auth Flow

Registration

The user provides a name, email, and password. The server generates a unique account number and wallet ID using node:crypto, hashes the password with bcrypt, and inserts the record into PostgreSQL.

Login

The user provides their account number and password. The server fetches the stored hash, runs bcrypt.compare(), and on success signs a JWT containing the user's ID and account number. The token is set as an HttpOnly cookie with the Secure flag.

Protected Routes

Every protected route passes through an authMiddleware function that reads the cookie, verifies the JWT signature using the secret key, and attaches the decoded payload to the request. Requests with missing or invalid tokens receive a 401.

Database Schema

users

CREATE TABLE users (
  id           SERIAL PRIMARY KEY,
  name         VARCHAR(100) NOT NULL,
  email        VARCHAR(100) UNIQUE NOT NULL,
  password     TEXT NOT NULL,              -- bcrypt hash
  account_no   VARCHAR(20) UNIQUE NOT NULL,
  wallet_id    VARCHAR(20) UNIQUE NOT NULL,
  balance      NUMERIC(12,2) DEFAULT 0.00,
  status       VARCHAR(10) DEFAULT 'active', -- 'active' | 'frozen'
  created_at   TIMESTAMPTZ DEFAULT NOW()
);

transactions

CREATE TABLE transactions (
  id           SERIAL PRIMARY KEY,
  user_id      INT REFERENCES users(id) ON DELETE CASCADE,
  type         VARCHAR(20) NOT NULL, -- 'deposit' | 'withdrawal' | 'transfer'
  amount       NUMERIC(12,2) NOT NULL,
  note         TEXT,
  created_at   TIMESTAMPTZ DEFAULT NOW()
);

Auth Endpoints

POST /api/register

Register a new user account.

// Request body
{
  "name":     "Jane Doe",
  "email":    "jane@example.com",
  "password": "supersecret"
}

// Response 201
{
  "message":    "Account created.",
  "account_no": "VV-83920184",
  "wallet_id":  "WLT-4A2F9C"
}

POST /api/login

Authenticate and receive a session cookie.

// Request body
{
  "account_no": "VV-83920184",
  "password":   "supersecret"
}

// Response 200 — sets HttpOnly cookie: token=<jwt>
{ "message": "Login successful." }

POST /api/logout

Clears the session cookie. No body required.

Account Endpoints

GET /api/account

Returns the authenticated user's account details and current balance.

// Response 200
{
  "name":       "Jane Doe",
  "email":      "jane@example.com",
  "account_no": "VV-83920184",
  "wallet_id":  "WLT-4A2F9C",
  "balance":    "2500.00",
  "status":     "active",
  "created_at": "2026-05-01T10:00:00Z"
}

Transaction Endpoints

POST /api/deposit

// Request body
{ "amount": 500 }

// Response 200
{ "message": "Deposit successful.", "new_balance": "3000.00" }

POST /api/withdraw

// Request body
{ "amount": 200 }

// Response 200
{ "message": "Withdrawal successful.", "new_balance": "2800.00" }

POST /api/transfer

// Request body
{
  "recipient": "recipient@example.com", // email OR wallet_id
  "amount": 150
}

// Response 200
{ "message": "Transfer successful.", "new_balance": "2650.00" }

GET /api/transactions

Returns a list of all transactions for the authenticated user, ordered by most recent.

// Response 200
[
  {
    "id": 12,
    "type": "deposit",
    "amount": "500.00",
    "note": null,
    "created_at": "2026-05-15T14:22:00Z"
  },
  ...
]

Admin Endpoints

All admin routes require the X-Admin-Token header, which is validated server-side against the ADMIN_PASSWORD environment variable.

GET /api/admin/users

Returns all registered users with their balances and statuses.

POST /api/admin/freeze/:id

Sets the user's status to frozen. Frozen users cannot log in.

POST /api/admin/unfreeze/:id

Restores the user's status to active.

DELETE /api/admin/user/:id

Permanently deletes the user and all associated transaction records.

GET /api/admin/transactions/:id

Returns the full transaction history for a specific user by their ID.
