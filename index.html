Why build a bank?

I wanted a project that forced me to think about routing, auth, database design, and concurrency all at once. Banking does that. Every feature has real stakes: a transfer that partially succeeds is worse than one that fails cleanly.

⚙

No Framework

Routing, body parsing, and response headers are all handled on Node's native http module. Writing this without Express taught me what Express is actually solving, which makes me better at using it.

🔒

Auth Done Right

Passwords hashed with bcrypt at cost 10, sessions via stateless JWTs in HttpOnly cookies. Tokens stay out of JavaScript entirely. Frozen and soft-deleted accounts are rejected at every protected route.

🗄

Three-Table Schema

Users, wallets, and transactions are separate tables with proper foreign keys and constraints. Balances live on wallets, not users. No ORM, just parameterized SQL queries.

💳

Atomic Transfers

Fund transfers run inside a PostgreSQL transaction with FOR UPDATE row locks. Either both sides commit or neither does. No partial state, no race conditions.

🛡

Separate Admin Identity

The admin panel has its own login flow and its own JWT cookie, separate from user sessions. Admin routes check both the token signature and the role claim before doing anything.

🧪

Isolated Transaction Layer

Payment processing is intentionally out of scope. The focus is the routing, the database writes, and the auth layer. You can deposit any amount and the ledger behaves correctly, which is the point.
